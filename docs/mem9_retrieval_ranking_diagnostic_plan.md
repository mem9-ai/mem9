## Why This Is Not A Retried Abandoned Strategy

- Not broad keyword/BM25 admission: the rule does not fetch or admit new lexical candidates; every candidate must already be in the server ranked candidate list.

- Not broad entity boosting: entity fields may remain present as existing trace evidence, but the proposal does not add generic entity boosts.

- Not source-packet/source-tail expansion: it does not append source turns or child artifacts; it only uses existing provenance metadata to decide final selection among already retrieved memories.

- Not broad temporal/date boosting: temporal proximity is treated as a distractor pattern, not a global date boost.

- Not benchmark-side repair: no answer prompt, scoring, category, or question-id logic changes.

## Exact Hypothesis

For a large homogeneous subset of Cat1/Cat4 ER0 failures, mem9 already retrieves the source-linked evidence memory into raw and ranked candidates, but the final selector drops it because confidence/coverage balancing favors session turns or semantically adjacent distractors. A narrow final-selection provenance rescue, limited to already retrieved candidates and capped at one replacement per speaker recall, will increase final-context evidence recall for those rows without increasing context size or changing ingestion.

## Exact Minimal Implementation Scope

Add a narrow final-selection rescue inside `server/internal/handler/recall.go`, after existing candidate retrieval, confidence assignment, dedupe, and current selection. The rule should only consider candidates already present in the server's ranked candidate list, with explicit source provenance (`seq`, `source_seqs`, or `source_turns`), adequate confidence, and raw/RRF rank evidence. It may replace at most one low-coverage selected memory per speaker recall with a source-provenance-bearing candidate that is currently just below final top-k. No new candidates are admitted and the returned limit remains unchanged.

Initial guardrails for the approved implementation should be conservative: candidate must be in ranked candidates, have explicit source provenance, have confidence at least 65, have raw/RRF rank in the observed support band, and replace only the weakest selected memory that adds no unique query coverage. Low-confidence subtype A is deliberately not the primary target.

## Tests To Add Before Benchmark

- Unit test: high-confidence source-provenance candidate ranked 11-20 replaces a lower-coverage selected memory, keeping returned count fixed.

- Unit test: candidate below confidence 65 is not rescued by default.

- Unit test: no rescue when candidate lacks `seq`, `source_seqs`, or `source_turns` provenance.

- Unit test: no rescue when all selected memories contribute unique query coverage and candidate is redundant.

- Regression test: exact/time query source-pool preferences remain unchanged unless the rescue guard is explicitly satisfied.

## Trace-Only Validation Plan

1. Build an offline replay over the 3972 trace JSON files that simulates the proposed final-selection rescue without changing server code.

2. Primary pass criterion: at least 8 safe target rows move a gold dia_id into simulated final top-k with at most 4 risk rows. The current evidence-backed opportunity is 157 safe rows / 4 risk rows.

3. Inspect removed competitors for the first 30 rescued rows to confirm the rule removes redundant or low-coverage context, not unique evidence.

4. If implementation is later approved, run the focused Go tests and one predict-only trace capture, not a full benchmark, to prove normal output shape and trace lineage.

## Full Benchmark Validation Lane, Not To Run Yet

After explicit approval and local trace validation only: run the accepted mem0-style settings with fresh clean cache if the implementation touches ranking, then run full LoCoMo once. Settings must remain `LOCOMO_PROTOCOL=mem0-style`, `LOCOMO_ANSWER_PROMPT=mem0-speaker`, `LOCOMO_SPEAKER_INGEST_MODE=batch2`, `LOCOMO_SPEAKER_TOP_K=10`, and `MEM9_RETRIEVAL_LIMIT=20`. Overall LLM is secondary; row-level evidence movement is primary.

## Expected Metric Movement

Primary expected movement: increased Cat1/Cat4 evidence recall and final-context gold dia_id presence on the 157 safe rows in trace replay. Full benchmark secondary expectation is modest positive Overall LLM movement only if answer synthesis uses the newly present evidence; do not use one Overall LLM run as proof.

## Success Criteria

- Trace replay: one production-legitimate rescue mechanism covers >=8 safe rows with <=4 risk rows.

- Focused tests pass and returned memory count stays capped.

- Predict-only validation shows rescued evidence appears in final context for target rows without broad context growth.

- Full benchmark, if later approved, shows Cat1/Cat4 evidence recall improvement without large regressions in other categories or speaker buckets.

## Failure Criteria

- Trace replay gains are heterogeneous or depend on question/category IDs.

- More than 4 risk rows are rescued by the same mechanism.

- The rule admits new candidates, expands context, or behaves like a generic keyword/entity/date/source boost.

- Focused tests show low-confidence or provenance-free candidates are rescued.

## Rollback Plan

Keep the implementation in one helper or one small selection block in `server/internal/handler/recall.go`; rollback is reverting that block and its tests. Trace-only instrumentation should remain separate and can be reverted independently. No benchmark or harness files should change.

## Overfitting Risk

Medium. The evidence is row-level and mechanism-specific, but it comes from LoCoMo traces. Guardrails reduce overfitting: no question/category IDs, no prompt/scoring changes, no new candidate admission, fixed top-k, provenance requirement, confidence floor, and trace replay before full benchmark.
